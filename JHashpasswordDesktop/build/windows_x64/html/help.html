How I came to write this program

I got the idea to write this program from an entry in Heise News of 2004-08-09.

Most people can't remember a great count of different passwords for different purposes. So they use the same password for different purposes (yes, me too... before HashPassword). This makes the password vulnerable for phishing. The usage of a password at a totally unsuspected site, makes it possible somebody abuses the password at a totally different site, where you sure never would like it (e. g. home banking). - Oops! Nasty situation! So, what?

A good idea would be to make a single password (better a passphrase; in the following passphrase stands for what you have to remember, password stands for what the host gets to see) somehow unique for different hosts. And this in a way, that knowing the one password even with all but the passphrase will not allow to draw a conclusion to the other. But one would still have to remember just a single passphrase.


How's that?

Hashing!

Hashing creates a one-way code of a string. One-way code means, that no-one can determine the passphrase string from the result but the same passphrase always creates the same result.

Hashing also ensures even the least modification of the input results in a totally different passwords. If we take your passphrase and add the (saved) host name and if applicable the (saved) login name, with hashing we get a totally different code for any host (and any login name). If we translate this code back to a string, this string has a good random pattern, because it matches the unpredictable hash code.

I achieve this by translating the code to a kind of 'number' of base N, where we use the defined (saved) character set with N characters as permitted 'digits'. That's easier than it sounds. If you like, you can learn how. If not just skip the blue explamnation part.

Aside: The Translation

Everybody has heard of division of integer values where the result is an integer again and the division remainder. (That is how we learn mathematics, don't we?) The remainder is always in the range 0...(N-1) if we divide by N.

Let us make it easy - let's say we divide by N=10 and the available characters for the possible remainder-s (numbers!) 0...9 are ordered to: "0123456789" (No! I'm not joking, just follow the explanation and see!). The hash value might be the number 7340. Okay, here we go:

    7340/10 = 734 R 0
This remainder value (a number!) 0 is now translated by the first character in the list (we always add a 1 to the remainder and that helps us to find the corresponding character in the lsit) this is the "0" (if I write 0 I speak of the number if I write "0" I speak of the "image" of that number) and will be the last character in the string (image) that describes the hash.

   734/10 = 73 R 4
This means we have to take the fiifth character in the list (the "4") and this is the last character but one in the string that describes the hash. So we have already "40".

   73/10 = 7 R 3
We take the fourth character in the list (the "3") and this is the next (temporary) leader in the string that describes the hash. So we have already "340".

   7/10 = 0 R 7
We take the eigth character in the list (the "7") and this is the final leader in the string that describes the hash, because the integer part of the quotient is 0. So we finally have "7340". You might think this is just the number we started, but no, it is just a string.

Usually we are working with the base 10 and the "character set" defined above to write down the numbers. But what we write down is not the number itself it is the "image" of a number with a certain set of characters that we all agreed to (or better have learned to agree to).

In that sense the 7340 and "7340" are identically: Both are images! I only have the chance to tell you a number by write such an image (or "explain" this image when I tell you: "seventhousand- and-threehundered-and-fourty").

If we use N=16 and the ordered character set: "0123456789ABCDEF" (exactly 16 characters as needed), we can write down the number 7340 as "1CAC" (try it yourself by dividing by 16 like I did in the example above with 10).

Now what happens if we have N=64 and the ordered character set is:
"0123456789ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz@_"? Can we do it with this set too? Sure! The result, the "image" noted with the base 64 and that set, of the number 7340 is "1oi" (mention, it is important to make a distinction between lower case and upper case characters!).

"7340(10)", "1CAC(16)" and "1oi(64)" are representations of the same number! (Well, to be 100% precise I would have to write down each used character set too, but for 10 and 16 I used the character sets that are commonly agreed on and for 64 it is just the one described above...) They stand for the same number, only the representation is chosen differently. In mathematics that is said as: "A number exists independently from its representation (image)."

I hope you enjoyed that excursion.



Voilá!

That is, what this program does: It takes your passphrase, which you can easy remember (it should be a well formed passphrase though; see e.g. 'diceware') adds the string of the host you defined to memorize and the login name you defined to memorize and creates a hash code (selectable MD5, SHA1, SHA256, SHA384, or SHA512) of the resulting string. This hash code (number) is translated using the character set you defined to a representation of that number and because the number is "quasi random" the string (the representation) is "quasi random" too, which is the real password and the program copies the result with the allowed length or if possible with the full (necessary) length (to achieve full correspondency) to the clipboard.

The length you define is an upper limit which might fall short by the full length created password! Even though the created password is equivalent to a quasi random number of 128 (MD5) or 160 (SHA1) or the namend count of bits (SHA###)  and as safe. If your defined upper length limit is less than the full length the resulting string's security is equivalent a quasi random number of <character_count_in_set><password_length> (e.g. 64 different characters in the set, 10 characters password length => 1,152,921,504,606,846,976 possible combinations - not too bad). You thus can even with a limited password length increase the safety by adding as much allowed characters to the character set as possible (if the host accepts others...). If your host accepts password lengths of 32 characters or more you can keep the predefined character set.

If you like you find a button "Charset Random Mix". If you use it, the character set is put to a random order. Be aware using it only once, when you prepare a new account (or if you want to create a new password for the account that way)! If you do it later, you will perhaps never be able to recreate you password! Same is of course true, if you alter the character set manually!


How - where - what?

All provider/host's name are stored to an XML file as you entered them. With that the defined maximum password length the character set and the hash type is stored, too. If you have more than one login at the same host, they are stored, too. The maximum password length the character set and the hash type is stored for each login separately. The real password will always be created at runtime using the passphrase you have to enter and is then copied to the clipboard. The real password will thus never be visible and is thus hard to spy (the entry of the passphrase should be secrete, though and the passphrase should be of good quality! ;-) ).

A real security feature is that whatever entered, a password is created. If an attacker "Cleo" has your XML-file and my (this) program (latter is easy), "Cleo" has to try every single password at the hoster's login. Usually the provider (hoster) will realize several bad passwords in a row and react by setting a lock for the account for a certain time (e. g. T-Online is doing so). That means a brute-force-attack might be lengthy. ;-)

Since version 3 of the program the used hash algorithm, the password length and the character set to each login is stored to the XML file individually. This make the program more flexible.

The actual version 4 adds hash-codes SHA256, SHA384 and SHA512 and the possibility to randomize the charset order. (Of course only the first time, if you want to create a new account - otherwise you will not get your password again. Because most users will just use the predefined 64 character set I wanted to add some variety because this way every account will be able to more or less use its own char set. There are about 1,268869x1089  permutations possible which makes an attack without your XML-file even harder.)

The most important reason for version 4 is the move to the free version of Borland Turbo C++ 2006. This way really everyone can translate my sources and check them. But please note my copyright! (Maybe after all I'm publicity-addicted. >:->)

Some programs do not allow taking passwords from the clipboard (as far as I know, that is not with web browsers and the web, but you might like to use this program elsewhere, too...). Then you have a button to show the password in a message box and type it in manually. But be aware, that the real password can be compromised then, e. g. if s. o. is peeking over your shoulder! Type in the real password as it is, case sensitive!

Theoretically you can use my program and the XML-file from a USB memory stick even in an internet cafe. But I suggest to be carefully: If the computer is manipulated it doesn't help, that my program is secrue (as far as I know). If all of your data from the XML-file and your passphrase got phished, your eBay account might show some strange activity after. This is not my fault... Of course the phishing thread is always there even without using my program. I just wanted to underline even this program requires a safe computer!

Though I still know no way to crack, and I think it is not possible with today computers with the knowledge of your XML-file and a small count aptured hashcodes (passwords) to compute socalled "collisions" to create a so far unknown password, it might be not impossible.
